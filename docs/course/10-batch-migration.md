# 10 — Build the Batch Client Extension and Verify Headless Batch Engine

## Goal

Package repeatable landing-page content migration using a real Batch Client Extension and version-generated Batch Engine payload.

## Separation of concerns

| Browser importer | Batch Client Extension |
|---|---|
| User selects Excel and assets | Deployment packages `*.batch-engine-data.json` |
| Runs in signed-in browser | Runs through Batch Engine and OAuth server |
| Validates editorial input | Seeds or transfers reviewed data |
| Calls Documents and Structured Content APIs | Uses Batch Engine-generated `jsont` contract |

## BE tasks

1. Confirm all expected `NXC-*` content is published.
2. Export required DTOs through Headless Batch Engine using `jsont`.
3. Poll export task until completion.
4. Download and extract the export.
5. Preserve the generated `configuration` block.
6. Filter items by approved ERC prefix and reject duplicates.
7. Package payloads in dependency order.
8. Deploy the Batch Client Extension twice.
9. Record import counts and failed-item details.

## Required API flow

```text
POST /o/headless-batch-engine/v1.0/export-task/{className}/jsont?siteId={site}
GET  /o/headless-batch-engine/v1.0/export-task/{taskId}
GET  /o/headless-batch-engine/v1.0/export-task/{taskId}/content
```

Where supported for direct import verification, record the matching import task, poll its state, and inspect failed items through the target version's API Explorer.

## Package order

A clean target may require multiple numbered payloads:

```text
10-documents.batch-engine-data.json
20-structured-content.batch-engine-data.json
```

Structures and Templates remain explicit target prerequisites when the running Classic Web Content APIs do not expose supported portable create operations.

## Client Extension contract

```yaml
nexcent-content-batch:
    name: Nexcent Landing Content Batch
    oAuthApplicationHeadlessServer: nexcent-content-batch-oauth
    type: batch
```

The OAuth application must include Batch Engine and Headless Delivery scopes.

## Verification

- Payload uses the generated DTO class name and `jsont` format.
- Every packaged item uses an approved `NXC-` ERC.
- Unrelated site content is excluded.
- First deployment creates/imports expected data.
- Second deployment does not create duplicate ERCs.
- Missing Structure or media dependencies fail with documented actionable errors.

## Checkpoint

- [ ] Payload is generated by the running Liferay version.
- [ ] Configuration block is not invented manually.
- [ ] Batch archive contains ordered files under `batch/`.
- [ ] OAuth registration and scopes are valid.
- [ ] Repeat deployment is demonstrated.
